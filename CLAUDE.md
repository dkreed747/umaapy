# CLAUDE.md

Developer guide for AI assistants working on the UMAAPy codebase.

## Project Overview

UMAAPy is a Python SDK for building UMAA (Unmanned Maritime Autonomy Architecture)-compliant applications over DDS (Data Distribution Service) middleware. It provides high-level abstractions for UMAA multi-topic graphs, report/command patterns, and message composition helpers on top of DDS middleware.

- **PyPI**: `umaapy`
- **Python**: 3.13+
- **Homepage**: https://github.com/dkreed747/umaapy
- **Docs**: https://dkreed747.github.io/umaapy/

## Strategic Direction — Cyclone-First Migration

**The project is actively migrating away from RTI Connext DDS to CycloneDDS as the sole runtime backend.** RTI is a proprietary vendor SDK requiring a paid license; CycloneDDS is open source (Apache 2.0) and removes all commercial toolchain dependencies.

This is a breaking-change release tracked as **UMAAPy 2.0.0**. All new work must be oriented toward the Cyclone-first architecture:

- **Do not add new RTI Connext DDS dependencies or deepen RTI coupling.** The current `rti.connext>=7.5.0` in `pyproject.toml` is being removed as part of P2.
- **Do not add RTI imports to any new source files.** Existing RTI imports in `src/umaapy/core/` and `src/umaapy/util/dds_configurator.py` are being actively replaced.
- **Prefer the middleware port/adapter pattern.** The P1 epic established middleware port contracts and a fake adapter for unit testing. New code should depend on port interfaces, not on `rti.connextdds` directly.

### Architecture Target (P2+)

The target architecture uses a **domain/port/adapter** layering:

```
core/ (domain)   →   middleware port interfaces   →   CycloneMiddlewareAdapter (runtime)
                                                  →   FakeMiddlewareAdapter (unit tests)
```

- **Domain modules** (`core/report_provider.py`, `core/report_consumer.py`, etc.) depend only on port interfaces, not on any specific DDS SDK.
- **Middleware ports** (contracts added in P1) define the participant, topic, reader, writer, and listener abstractions.
- **CycloneMiddlewareAdapter** (P2 work, `src/umaapy/`) implements the ports using `cyclonedds`.
- **FakeMiddlewareAdapter** (P1 work) implements the ports in-process for deterministic unit tests.

## Development Setup

```bash
# RTI-free development (preferred going forward)
pip install -e .[tests,cyclone]
UMAAPY_AUTO_INIT=0 python -c "import umaapy; print(umaapy.__file__)"
pytest -m "not integration_vendor"

# With RTI Connext DDS (legacy full-stack integration tests only)
# Requires valid rti_license.dat and rti.connext installed
pip install -e .[tests]
pytest
```

The package must be installed in editable mode (`pip install -e .`) — the test suite enforces this via `conftest.py`.

**Note on current dependencies**: `pyproject.toml` currently lists `rti.connext>=7.5.0` as a core dependency. This is being removed as part of P2 migration (#26–28). When that work is complete, `cyclonedds` will become the sole runtime dependency and the `[cyclone]` extra will be folded into the base install.

## Running Tests

```bash
# Default: exclude vendor integration tests (no RTI license required)
pytest -m "not integration_vendor"

# Equivalent shortcut
make test

# All tests (requires RTI Connext DDS license at $RTI_LICENSE_FILE or $NDDSHOME/rti_license.dat)
pytest
```

### Test Markers

| Marker | Description |
|---|---|
| `unit` | Fast, isolated, no live DDS middleware |
| `component` | Multi-module tests, no vendor middleware |
| `integration_vendor` | Requires RTI Connext DDS runtime and a valid license |
| `integration_cyclone` | *(planned, P2 #29)* Requires CycloneDDS runtime in a container |

The `conftest.py` automatically skips `integration_vendor` tests when no valid RTI license is detected. It also installs stub modules for `rti.connextdds`, `rti.idl`, and `rti.rpc` so unit/component tests can import the package without RTI installed.

The stub injection in `conftest.py` is a **transitional mechanism** — as P1/P2 work decouples `core/` from RTI imports, these stubs will no longer be needed and will be removed.

## Generated Types — Critical Rules

`src/umaapy/UMAA/` is entirely machine-generated from IDL files using the CycloneDDS `idlc` compiler. **Do not edit these files manually.**

`src/umaapy/umaa_types.py` is a curated re-export façade that surfaces selected generated types under clean names. Edit it carefully — it is excluded from Black formatting.

### Regenerating Types

Requires CycloneDDS `idlc` compiler (available from the `cyclonedds-nightly` package):

```bash
# Incremental regeneration
python scripts/generate_types.py

# Clean regeneration (removes UMAA/ first)
python scripts/generate_types.py --clean
make generate-types-clean
```

The script:
1. Discovers all `*.idl` files under `specs/idls/`
2. Runs `idlc -l py` on each file, outputting into `src/umaapy/UMAA/`
3. Patches the generated `UMAA/__init__.py` to alias `UMAA` → `umaapy.UMAA` for cross-import compatibility

**After regeneration**: commit changes under `src/umaapy/UMAA/`. CI runs a drift check on `windows-latest` that fails if generated output differs from what is committed.

The pinned version is `cyclonedds-nightly==2025.11.25`. Do not upgrade without updating the CI drift check job and validating generated output.

## Key Abstractions

### DDSConfigurator (`umaapy.util.dds_configurator`)

Singleton managing the DDS DomainParticipant, Publisher, Subscriber, and QoS profiles. Thread-safe via a class-level lock. **Currently RTI-backed; being replaced by CycloneMiddlewareAdapter in P2.**

```python
from umaapy import get_configurator, reset_dds_participant

cfg = get_configurator()
reader = cfg.get_reader(SomeType)                    # plain DataReader
writer = cfg.get_writer(SomeType)                    # plain DataWriter
umaa_reader = cfg.get_umaa_reader(SomeType)          # UMAA multi-topic reader graph
umaa_writer = cfg.get_umaa_writer(SomeType)          # UMAA multi-topic writer graph

reset_dds_participant()   # tears down and recreates the participant (useful in tests)
```

QoS profile categories: `UmaaQosProfileCategory.COMMAND`, `.CONFIG`, `.REPORT`.

### EventProcessor (`umaapy.util.event_processor`)

Thread-pool singleton for dispatching DDS listener callbacks and UMAA commands asynchronously. Priorities: `LOW`, `MEDIUM`, `HIGH`.

### UMAA Concepts (`umaapy.util.umaa_utils.UMAAConcept`)

The SDK classifies IDL types by which UMAA fields they contain:

| Concept | Key fields |
|---|---|
| `REPORT` | `timeStamp`, `source` |
| `COMMAND` | `timeStamp`, `source`, `destination`, `sessionID` |
| `ACKNOWLEDGEMENT` | `timeStamp`, `source`, `sessionID`, `command` |
| `STATUS` | `timeStamp`, `source`, `sessionID`, `commandStatus`, … |
| `GENERALIZATION` | `specializationTopic`, `specializationID`, `specializationTimestamp` |
| `SPECIALIZATION` | `specializationReferenceID`, `specializationReferenceTimestamp` |
| `LARGE_SET` | `setID`, `updateElementID`, `updateElementTimestamp`, `size` |
| `LARGE_SET_ELEMENT` | `element`, `setID`, `elementID`, `elementTimestamp` |
| `LARGE_LIST` | `listID`, `updateElementID`, `startingElementID`, `size`, … |
| `LARGE_LIST_ELEMENT` | `element`, `listID`, `elementID`, `elementTimestamp`, `nextElementID` |

### Multi-topic Graphs

The SDK automatically builds reader/writer graphs for complex UMAA topic patterns:
- **Generalization/Specialization**: one root topic + N specialization topics
- **Large Sets**: root topic + per-element set-element topics
- **Large Lists**: root topic + per-element list-element topics

`DDSConfigurator.get_umaa_reader()` / `get_umaa_writer()` construct and wire these graphs. The resulting adapter objects satisfy the `dds.DataReader` / `dds.DataWriter` interface.

### Core Providers/Consumers

| Class | Role |
|---|---|
| `ReportProvider` | Publishes stamped UMAA reports; implements `DataWriterListener` |
| `ReportConsumer` | Subscribes to UMAA reports; implements `DataReaderListener` |
| `CommandProvider` | Receives UMAA commands (content-filtered by `destination`); implements `DataReaderListener` |
| `CommandConsumer` | Sends UMAA commands |

All four stamp messages with `source` and `timeStamp` automatically.

## Common Pitfalls

- **Auto-init in tests**: Always set `UMAAPY_AUTO_INIT=0` in test environments or before importing `umaapy`. The `conftest.py` does this via `os.environ.setdefault`, but manual scripts must do it explicitly.
- **Editable install required**: Tests will abort with a clear error if `umaapy` resolves outside the local `src/` tree. Run `pip install -e .[tests]`.
- **UMAA/ is generated**: Any manual change to `src/umaapy/UMAA/` will be overwritten on the next type regeneration and will break the CI drift check.
- **Black exclusions**: Do not run Black on `umaa_types.py` or anything under `UMAA/`. The `pyproject.toml` `extend-exclude` handles this, but be aware when using `--include` overrides.
- **Topic names**: `topic_from_type(SomeType)` converts `_` separators in the class name to `::`, matching UMAA DDS topic naming conventions.
- **RTI license detection**: `conftest.py` checks `$RTI_LICENSE_FILE` and `$NDDSHOME/rti_license.dat`. Without a valid license, all `integration_vendor` tests are skipped automatically.
- **Do not add RTI imports**: Any new source file that imports from `rti.connextdds` moves in the wrong direction. New modules must depend on port interfaces. Existing RTI imports are being removed in P2 (#26–28).
- **Hidden RTI fallback**: When removing RTI imports, audit all transitive imports — a common mistake is removing the direct import but leaving an indirect path that still pulls in RTI. Run with `CONNEXT_AVAILABLE=False` logic or check via `import rti.connextdds` assertions in CI.
- **Cyclonedds version pin**: The type generation and runtime adapter are pinned to `cyclonedds-nightly==2025.11.25`. Do not change this without regenerating all types and updating CI.
